# Travel Planner 마스터 플랜

## 1. 활성 계획서 현황

| 상태 | 계획 | 폴더 | 남은 작업 | 예상 공수 |
| :--: | ---- | ---- | --------- | --------- |
| `완료` | 반응형 네비게이션 | `responsive-nav/` | 완료 | - |

## 2. 신규 기능 아이디어

### Tier 1 — 핵심 완성 (즉시~2주)

> MVP에서 빠진 CRUD 보완 + 코드 안정화

| ID | 기능 | 설명 | 예상 공수 |
| -- | ---- | ---- | --------- |
| **N-01** | 여행 삭제 | cascade 또는 soft delete, admin만 | 2~3시간 |
| **N-02** | 장소 수정 | place-form 재활용, 수정 모드 | 2~3시간 |
| **N-03** | 멤버 역할 변경/제거 | admin 전용 UI | 3~4시간 |
| **N-04** | 에러 핸들링 통일 | 공통 에러 바운더리 + API 응답 표준화 | 4~6시간 |
| **N-05** | 장소 카테고리 필터 | 탭 또는 드롭다운 | 1~2시간 |

### Tier 2 — 사용성 개선 (2~4주)

> 사용자 경험 대폭 향상

| ID | 기능 | 설명 | 예상 공수 |
| -- | ---- | ---- | --------- |
| **N-06** | AI 결과 → 일정 반영 | JSON 파싱 → schedule_items 자동 삽입 | 1~2일 |
| **N-07** | 지출 차트 시각화 | recharts 파이차트/바차트 | 3~4시간 |
| **N-08** | 여행 커버 이미지 | Supabase Storage 업로드 | 3~4시간 |
| **N-09** | OG 메타태그 | 소셜 공유 미리보기 | 2~3시간 |
| **N-10** | 디자인 시스템 정리 | 컬러/타이포/컴포넌트 규칙 문서화 | 1일 |

### Tier 3 — 출시 준비 (1~2개월)

> 배포, 테스트, 모니터링

| ID | 기능 | 설명 | 예상 공수 |
| -- | ---- | ---- | --------- |
| **N-11** | CI/CD 파이프라인 | GitHub Actions (빌드+린트+타입체크) | 3~4시간 |
| **N-12** | Vercel 배포 | 프로덕션 환경 구성 | 2~3시간 |
| **N-13** | 보안 점검 | RLS, CSP, 환경변수, SSRF | 1일 |
| **N-14** | Vitest 환경 구축 | Unit + Integration 테스트 | 1일 |
| **N-15** | Sentry 에러 트래킹 | 프론트엔드 + API 에러 수집 | 2~3시간 |
| **N-16** | 랜딩 페이지 | 서비스 소개 + 시작하기 | 1~2일 |
| **N-17** | 이용약관/개인정보 | 법적 필수 문서 | 3~4시간 |

### Tier 4 — 확장 (2개월+)

> 차별화 기능, 수익화 검토

| ID | 기능 | 설명 | 예상 공수 |
| -- | ---- | ---- | --------- |
| **N-18** | 환율 자동 변환 | 실시간 환율 API 연동 | 1일 |
| **N-19** | 푸시 알림 (Web Push) | 멤버 활동 알림 | 1~2일 |
| **N-20** | 여행 템플릿 | "도쿄 3박4일" 등 인기 템플릿 | 1주 |
| **N-21** | 여행 공개/탐색 | 다른 사용자의 여행 계획 탐색 | 1~2주 |
| **N-22** | 날씨 연동 | 여행 기간 날씨 예보 표시 | 2~3일 |
| **N-23** | 항공편 연동 | 항공편 검색 + 일정 자동 추가 | 2~3주 |

## 3. 실행 로드맵

### Phase 0 — 기반 다지기 (0~2주)

> 코드 안정화 + 개발 환경 정비. "안심하고 기능을 추가할 수 있는 상태" 달성.

- [ ] **N-04** 에러 핸들링 통일 (4~6시간)
- [ ] ESLint + Prettier 설정 (2시간)
- [ ] Git 브랜칭 전략 + 커밋 컨벤션 수립 (1시간)
- [ ] Supabase 타입 자동 생성 전환 (1시간)
- [ ] console.log 정리 (1시간)

### Phase 1 — CRUD 완성 (2~4주)

> 빠진 기능 채우기. "모든 기본 동작이 가능한 상태" 달성.

- [ ] **N-01** 여행 삭제 (2~3시간)
- [ ] **N-02** 장소 수정 (2~3시간)
- [ ] **N-03** 멤버 역할 변경/제거 (3~4시간)
- [ ] **N-05** 장소 카테고리 필터 (1~2시간)
- [ ] **N-06** AI 결과 → 일정 반영 (1~2일)

### Phase 2 — UX 개선 + 출시 준비 (1~2개월)

> 디자인 완성 + 테스트 + 배포. "사용자에게 공개할 수 있는 상태" 달성.

- [ ] **N-10** 디자인 시스템 정리 (1일)
- [ ] **N-07** 지출 차트 (3~4시간)
- [ ] **N-08** 커버 이미지 업로드 (3~4시간)
- [ ] **N-09** OG 메타태그 (2~3시간)
- [ ] **N-11** CI/CD (3~4시간)
- [ ] **N-12** Vercel 배포 (2~3시간)
- [ ] **N-13** 보안 점검 (1일)
- [ ] **N-14** Vitest 테스트 (1일)
- [ ] **N-15** Sentry (2~3시간)

### Phase 3 — 공개 출시 (2~3개월)

> 랜딩 페이지 + 법적 문서 + 사용자 유입.

- [ ] **N-16** 랜딩 페이지 (1~2일)

- [ ] Vercel Analytics 연결 (1시간)
- [ ] 사용자 피드백 채널 구축 (2~3시간)

### Phase 4 — 확장 (3개월+)

> 차별화 기능으로 사용자 유지/확대.

- [ ] **N-18** 환율 변환

- [ ] **N-20** 여행 템플릿

## 4. 우선순위 판단 기준

| 기준 | 가중치 | 설명 |
| ---- | ------ | ---- |
| 사용자 가치 | 40% | 실제 여행 계획에서 필요한 기능인가 |
| 출시 차단 여부 | 30% | 이것 없이 공개할 수 있는가 |
| 구현 용이성 | 20% | 기존 코드/인프라를 활용할 수 있는가 |
| 차별화 | 10% | 경쟁 앱에 없는 기능인가 |
